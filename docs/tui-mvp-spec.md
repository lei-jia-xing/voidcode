# TUI MVP 规范

源 issue：#18

本文档定义了 VoidCode TUI MVP 的最低交互模型和规范冒烟流程。

本规范有意保持精简。目标是指定一个终端客户端，通过以键盘为主的交互、由运行时拥有的审批、有序的事件渲染以及持久化的会话重放来驱动现有的运行时边界。

## 与当前 CLI 的关系

TUI **不**取代现有的 CLI 契约工作，也**不**通过直接调用工具来绕过它。

对于 MVP，TUI 应被视为一个**新的运行时客户端**，与目前的文本 CLI 并列：

- CLI 仍然是有效的文本优先客户端和验证界面
- TUI 消费与 CLI 相同的运行时概念
- 两个客户端都依赖运行时进行执行、事件、审批和持久化

## 布局：启动页与会话页

TUI 采用了两阶段的极简布局，旨在通过流式对话驱动任务：

1. **启动页 (StartupView)**：默认启动且无 `--session-id` 时显示。
   - **单主容器**：居中的 VoidCode 标题，下方紧跟一个极简的提示词输入框。
   - **引导说明**：底部显示快捷键提示（`Enter` 提交，`Ctrl+J` 换行，`Ctrl+Q` 退出）。
   - **无干扰设计**：此页面不显示时间线、审批状态或会话列表，确保用户聚焦于开启新任务。
2. **会话页 (SessionView)**：提交首个提示词后或通过 `--session-id` 恢复会话时显示。
   - **紧凑状态头**：顶部显示会话 ID 和当前运行状态（Idle, Running, Waiting）。
   - **核心时间线 (Timeline)**：占据屏幕主体，用于渲染流式事件和工具活动。
   - **智能显隐**：输出区域和审批摘要仅在有内容或待审批时显示。
   - **常驻输入条**：底部固定提示词输入框。
3. **沉浸式背景**：
   - 采用终端原生背景策略 (`background: terminal`)，支持透明终端背景。
   - 界面元素使用低透明度的中性色表面，减少视觉干扰。

## 最低交互模型

TUI MVP 应支持以下交互能力。

### 1. 多行提示词输入

- 使用 `TextArea` 构建的智能编辑器，支持多行输入。
- **提交逻辑**：直接按 `Enter` 提交（非空文本）。
- **换行逻辑**：`Ctrl+J` 保证在任何终端环境下都能插入换行符。`Shift+Enter` 或 `Alt+Enter` 作为别名提供支持。
- **视觉反馈**：通过输入框状态文字（Busy, Waiting, Idle）实时反馈运行状态，忙碌时自动禁用输入。

### 2. 活动时间线

- 主要的时间线窗格渲染有序运行时事件。
- **平滑过渡**：从启动页提交后自动无缝切换到会话页，并开始流式渲染。
- **空状态适配**：根据上下文（新任务开始 vs 历史会话恢复）显示不同的引导文字。
- **重放一致性**：恢复之前的会话时，完整重放存储的事件序列。

该时间线足以显示 `docs/contracts/runtime-events.md` 中记录的当前已实现的事件进展。

### 3. 工具活动分组

- 工具相关的事件在时间线中显示为分组活动块。
- **视觉层级**：分组块使用左侧色条标识状态（活动中、已完成、已失败），保持紧凑且可读。
- **自包含**：TUI 内部即可完整理解工具请求的起止和结果。

### 4. 审批交互

- **审批模态框 (ApprovalModal)**：当运行时需要审批时，自动弹出全屏覆盖的模态框，强制用户做出决策。
- **本地持久化提示**：会话页会显示醒目的审批摘要，指引用户关注待处理请求。
- **决策同步**：审批结果实时发回运行时，并更新时间线状态。

### 5. 会话恢复

- 支持通过 `--session-id` 直接启动并打开特定的会话页面。
- 启动后自动跳过启动页，直接进入会话页重放历史。
- 如果该会话正处于等待审批状态，启动后会立即自动弹出审批模态框。

### 6. 键盘优先的焦点模型

- 核心操作 100% 键盘驱动，不依赖鼠标。
- **智能导航**：`Tab` 键在核心元素间流转，系统会根据页面状态（如审批弹窗）自动管理初始焦点。
- **快速退出**：`Ctrl+Q` 全局绑定用于立即退出应用。



本文档不锁定最终的键位绑定。它仅要求以键盘为主的流程。

## 运行时边界规则

TUI MVP 必须遵守以下规则：

1. **运行时作为系统边界**
   - TUI 与运行时拥有的操作进行通信，包括 运行、列出、加载、恢复 和 审批解析
   - TUI 不直接调用工具

2. **事件驱动渲染**
   - 时间线状态派生自有序的运行时事件和最终输出
   - TUI 不应发明无法从持久化的运行时数据恢复的私有状态转换

3. **持久化拥有的重放**
   - 会话历史来自运行时持久化
   - 恢复会话必须重放持久化的运行时状态，而非从即时的客户端缓存重建 UI 历史

4. **审批所有权保留在运行时**
   - TUI 捕获用户意图并将审批决策发回运行时
   - 运行时负责验证并应用该决策

## 实时流式传输与重放预期

TUI MVP 必须在两种模式下支持相同的观测模型：

- 活动运行期间的**实时活动**
- 恢复之前会话时的**持久化重放**

事件渲染的含义应在两种模式下保持一致。恢复的会话不应需要单独的、以不同方式解释同一运行时历史的客户端逻辑。

这遵循了 `docs/contracts/stream-transport.md` 中的预期。

## 规范冒烟流程

该冒烟流程是证明 TUI 对于 MVP 工作可用的最低演示。

### 流程

1. 启动 TUI 时不带会话 ID，进入启动页。
2. 输入提示词（如 `read README.md`）并按 Enter。
3. 界面自动切换到会话页，观察 `Timeline` 中的实时事件流：`request_received`、`tool_call`（分组块）和最终的回复输出。
4. 退出后，通过 `voidcode tui --session-id <id>` 重新打开。
5. 验证应用绕过启动页直接进入会话页，历史记录完整重放。
6. 针对写操作（如 `write test.txt "hello" --approval-mode ask`），验证 `ApprovalModal` 自动弹出，且可在模态框内批准或拒绝。

### 预期可观测结果

- 操作员可以在不离开 TUI 的情况下提交请求
- 操作员在工作进行时看到有序的运行时进度，而不是冻结的终端
- 需要审批的执行会显式暂停，并可以在 TUI 内部解决
- 存储的会话可以再次被找到，并使用运行时支持的状态重放
- 重放的历史保留了原始可观测流的含义和顺序

## 验收标准

- TUI MVP 规范已存在，且足够具体，可以无歧义地切分后续实现 issue
- TUI 被定义为一个运行时客户端，而非直接的工具执行器
- 最低交互集是明确的：提示词输入、时间线渲染、分组工具活动、审批处理，以及通过 `--session-id` 的会话恢复
- 规范冒烟流程覆盖了一个只读请求、一个需要审批的请求以及持久化的会话重放
- TUI 行为与运行时拥有的审批、运行时事件顺序和运行时持久化契约一致
- MVP 保持键盘优先，核心任务的完成不依赖鼠标交互或 shell 退回

## 非目标

本规范**不**定义：

- 深度的视觉磨合、主题或动画工作
- 多智能体终端 UX
- 新的事件模式、审批模式或有线协议
- 完整的 TUI 内部配置编辑器
- 除 MVP 流程所需之外的高级终端多路复用

## 后续实现切片

至少，本规范之后的实现工作应能被切分为以下独立的 issue：

- TUI 外壳与焦点管理
- 基于运行时事件的活动时间线渲染
- 分组工具活动渲染
- 审批交互处理
- 直接会话恢复流（`--session-id`）
- TUI 冒烟或端到端验证
